💾 SQL: Compliance Risk Query

▼
SQL-- Find contractors with the highest risk profile WITH complaint_summary AS ( SELECT licence_id, COUNT(*) AS total_complaints, COUNT(CASE WHEN complaint_date >= DATEADD(month, -12, CURRENT_DATE) THEN 1 END) AS complaints_last_12m, COUNT(CASE WHEN severity = 'HIGH' THEN 1 END) AS high_severity_count FROM complaints GROUP BY licence_id ), financial_health AS ( SELECT licence_id, net_tangible_assets, current_ratio, LAG(net_tangible_assets) OVER ( PARTITION BY licence_id ORDER BY reporting_period ) AS prev_nta, CASE WHEN net_tangible_assets < required_minimum_nta THEN 1 ELSE 0 END AS below_mfr FROM financial_reports WHERE reporting_period = (SELECT MAX(reporting_period) FROM financial_reports) ) SELECT l.licence_id, l.business_name, l.trade_category, c.complaints_last_12m, f.current_ratio, f.below_mfr, (COALESCE(c.complaints_last_12m,0)*2 + COALESCE(f.below_mfr,0)*5) AS risk_score FROM licensees l LEFT JOIN complaint_summary c ON l.licence_id = c.licence_id LEFT JOIN financial_health f ON l.licence_id = f.licence_id ORDER BY risk_score DESC LIMIT 50;

Quick-Fire Q&A

Click each question to reveal the answer.

Python / Data

How would you handle a dataset with 80% missing values in a key column?
First investigate why it's missing (MCAR/MAR/MNAR). Options: drop the column if not predictive; impute with median/mode for simple cases; use model-based imputation (KNN, iterative); create a binary "is_missing" indicator feature; or use models that handle missing values natively (XGBoost, LightGBM).
Explain data lake vs data warehouse
Data Lake: Raw, unstructured/semi-structured data, schema-on-read, cheap storage (S3), good for exploration. Data Warehouse: Structured, curated, schema-on-write (Snowflake), optimized for analytics queries, governed. Modern approach: lakehouse combines both.
How do you handle class imbalance in fraud detection?
SMOTE/ADASYN for oversampling; class weights in the model; threshold tuning (lower threshold = higher recall); ensemble methods; anomaly detection approach (treat fraud as outliers). In regulatory context, optimize for recall — better to flag false positives than miss fraud.

Machine Learning

What's the difference between L1 and L2 regularization?
L1 (Lasso): Adds absolute value of coefficients; produces sparse models (feature selection). L2 (Ridge): Adds squared coefficients; shrinks all weights but doesn't zero them. Use L1 when you suspect many irrelevant features; L2 when most features are useful.
How do you detect model drift in production?
Monitor: (1) Data drift — statistical tests (KS test, PSI) on input features; (2) Concept drift — track prediction distribution shifts; (3) Performance drift — monitor accuracy/recall on labeled feedback. Set up alerts when metrics cross thresholds. Schedule periodic retraining.
Explain precision vs recall in a regulatory context
Precision: Of contractors we flagged, how many were actually non-compliant? Recall: Of all non-compliant contractors, how many did we catch? In regulation, recall matters more — missing a non-compliant contractor (false negative) can harm consumers, while investigating a compliant one (false positive) just costs inspection time.

GenAI

Key risks of LLMs in government regulatory context?
Hallucination — generating plausible but false regulatory guidance. Data privacy — sensitive data in prompts. Bias — unfair treatment of certain groups. Accountability — who is responsible for AI-driven regulatory decisions? Mitigation: RAG for grounding, human-in-the-loop, audit logging, data anonymisation.
Fine-tuning vs RAG — when to use each?
RAG: When you need up-to-date knowledge, citations, or domain-specific docs — no model retraining needed. Fine-tuning: When you need the model to learn a specific style, format, or task (e.g., always output structured JSON). For QBCC: RAG for regulatory document Q&A; fine-tuning if you need a specialized complaint classifier.
How do you evaluate RAG system quality?
Three dimensions: (1) Retrieval relevance — are the right docs being retrieved? (2) Faithfulness — does the answer match retrieved context (no hallucination)? (3) Answer correctness — is the final answer accurate? Tools: RAGAS framework, human evaluation, A/B testing.

Responsible AI

How would you ensure a compliance model isn't biased?
Test model predictions across demographic slices (geography, trade type, business size). Use fairness metrics (equal opportunity, demographic parity). Apply SHAP for feature explainability — ensure no proxy discrimination. Regular bias audits. Document in model cards. Seek diverse stakeholder review.
What governance frameworks for AI regulatory recommendations?
Model cards (documenting purpose, limitations, performance); human-in-the-loop for high-stakes decisions; audit trail for every prediction; explainability requirements (SHAP); regular review cycles; compliance with Information Privacy Act 2009 and Public Sector Ethics Act 1994; risk-tiered approach (higher stakes = more oversight).
